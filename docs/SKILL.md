# Excalidraw Slides — AI-Generated Hand-drawn PPTX

Generate presentation slides with **AI-designed layouts**, hand-drawn aesthetics (rough.js + rough-notation), and visual theme presets — exported as `.pptx` files.

Every slide layout is uniquely generated by you. No fixed templates — you compose HTML/CSS/JS per slide for infinite visual variety.

## Trigger

Use when the user wants to create a presentation / PPT / slides / 簡報, especially with a hand-drawn or sketch style. Keywords: PPT, PPTX, slides, 簡報, presentation, hand-drawn slides, excalidraw slides.

## Prerequisites

- Python packages: `jinja2`, `playwright`, `python-pptx`, `lxml` (all pre-installed)
- Playwright browsers installed (`playwright install chromium`)
- Reference files in `.claude/skills/excalidraw-slides/`:
  - `DESIGN_SYSTEM.md` — Layout patterns, rough.js cookbook, annotation cookbook, data viz, element position templates
  - `STYLE_PRESETS.md` — 6 visual themes (Clean Sketch, Bold Marker, Notebook, Blackboard, Blueprint, Watercolor)
  - `render_slides.py` — Injects slide HTML into base template via Playwright to PNG (supports `--mode full|bg_only|both`)
  - `gen_pptx.py` — Assembles dual-layer PPTX (bg PNG + native editable elements) or legacy image-only PPTX

## 6-Phase Pipeline

```
Input (text / outline / PDF / markdown)
    ↓
Phase 1 — Content Structuring (split into 1-message-per-slide outline)
    ↓
Phase 2 — Style Selection (pick a style preset from STYLE_PRESETS.md)
    ↓
Phase 3 — Layout Planning (assign layout type per slide, enforce variety)
    ↓
Phase 4 — HTML Generation (compose HTML/CSS/JS per slide with DESIGN_SYSTEM.md)
    ↓
Phase 5 — Render + Visual Review (render_slides.py → read PNGs → check → fix)
    ↓
Phase 6 — Assemble (gen_pptx.py → .pptx)
```

---

## Phase 1: Content Structuring

Break input into a slide outline. Each slide = **one single message**.

Rules:
- Max 3-5 bullets per slide — prefer visual over text
- Use big numbers for impactful statistics instead of burying in bullets
- Use quotes for memorable phrases
- Insert section breaks for visual breathing room between topics
- If input is a PDF, read it first with the Read tool
- Ask the user for preferred language (English/Chinese) if not clear

### Visual Density Rules

- **60/40 rule**: At least 60% of each content slide's area must be visual (images, diagrams, rough.js illustrations, big numbers, charts). Max 40% text.
- **Headlines ≤ 8 words**: Force concise headings. Details go to speaker notes or supporting text, not slide headings.
- **Max 3 bullets**: If more than 3 points, split into multiple slides or use icon-grid/visual layout instead.
- **Every content slide must have a visual**: Either a concept illustration (rough.js), a placeholder image, a diagram, a chart, or a big-number display. No text-only slides except section-break/closing.

## Phase 2: Style Selection

Read `STYLE_PRESETS.md` and pick the preset that matches the content's mood.

| Preset | Best For |
|--------|----------|
| **Clean Sketch** (default) | Formal reports, paper reviews, corporate |
| **Bold Marker** | Keynotes, pitch decks, conference talks |
| **Notebook** | Reading notes, study materials, tutorials |
| **Blackboard** | Tech talks, live demos, developer audiences |
| **Blueprint** | Architecture diagrams, system design |
| **Watercolor** | Brand presentations, creative proposals |

Include the chosen preset's `style` JSON object at the deck's top level:

```json
{
  "style": { "bg": "...", "text": "...", "roughness": ..., "stroke_width": ... },
  "slides": [...]
}
```

## Phase 3: Layout Planning

Before generating any HTML, plan layout types across ALL slides to enforce variety.

**Available layout types** (reference patterns in `DESIGN_SYSTEM.md`):
- `centered-hero` — Title slide, centered
- `title-bullets` — Title + bullet list
- `two-column` — Two-column grid/split
- `three-column` — Three-column cards/grid
- `big-number` — Giant statistic + caption
- `quote` — Large quote block
- `timeline` — Horizontal timeline
- `process-flow` — Step boxes + arrows
- `icon-grid` — Emoji/icon grid (3-4 columns)
- `split-visual` — 50/50 or 60/40 image + text side-by-side
- `visual-hero` — Large rough.js concept illustration (center) + small text caption
- `comparison` — Side-by-side comparison with visual indicators
- `annotated-diagram` — Rough.js diagram with labeled callouts
- `full-bleed` — Giant centered text statement
- `section-break` — Section divider
- `closing` — Thank you / Q&A
- `diagram` — Mermaid.js diagram
- `chart` — Chart.js visualization
- `custom` — Freeform layout (for unique one-offs)

### Layout Selection Decision Tree

When choosing a layout for each slide, follow this priority order:

```
Has a key statistic?          → big-number
Has a quote?                  → quote
Has a process/steps?          → process-flow or timeline
Has comparison data?          → split-visual or comparison
Has a concept to explain?     → visual-hero (rough.js illustration + text)
Has a list of items (≤6)?     → icon-grid
Has a list of items (>6)?     → two-column or three-column with icons
Has a diagram/architecture?   → diagram (image_only) or annotated-diagram
Has an image/photo?           → split-visual (image + text side-by-side)
Default                       → split-visual (placeholder image + text)
```

The goal is **visual-first**: always choose the most visual layout that fits the content. Text-heavy layouts (`title-bullets`) should be the last resort, not the default.

**Variety rules:**
1. Never use the same layout type on consecutive slides
2. `title-bullets` may appear at most 2× in a 10-slide deck
3. At least 3 different layout types in any 5-slide window
4. Use at least 1 visual slide (big-number, quote, diagram, chart, icon-grid) per 4 content slides
5. Vary rough.js decoration patterns — don't repeat the same decoration on consecutive slides

Write out the layout plan as a comment before generating slides:
```
Layout plan: hero → two-column → section-break → big-number → title-bullets → ...
```

## Phase 4: HTML Generation

Read `DESIGN_SYSTEM.md` for layout patterns, rough.js cookbook, and annotation examples. Compose per-slide HTML/CSS/JS.

### JSON Schema

```json
{
  "style": {
    "bg": "#fffdf7", "text": "#1a1a2e", "text_muted": "#495057",
    "roughness": 1.0, "stroke_width": 1.5,
    "font_map": {
      "heading": "Ink Free",
      "body": "Ink Free",
      "annotation": "Ink Free",
      "cjk": "Microsoft JhengHei"
    }
  },
  "slides": [
    {
      "id": 1,
      "render_mode": "dual",
      "html": "<div>...your layout HTML...</div>",
      "css": ".your-class { ... }",
      "shapes_js": "drawRect(30, 30, 1860, 1020, {...});",
      "annotations_js": "annotateElement('el-id', {type:'underline'});",
      "accent_index": 0,
      "elements": [
        {
          "type": "textbox",
          "x": 160, "y": 350, "w": 1600, "h": 200,
          "paragraphs": [
            { "text": "Title", "font": "heading", "size": 54, "color": "#1864ab", "bold": true, "align": "center" }
          ]
        },
        {
          "type": "image",
          "x": 900, "y": 200, "w": 800, "h": 500,
          "src": null, "label": "[Insert photo here]"
        }
      ]
    }
  ]
}
```

### Slide Fields

| Field | Required | Description |
|-------|----------|-------------|
| `id` | Yes | Slide number (1-based) |
| `render_mode` | No | `"dual"` (default) — bg PNG + native elements. `"image_only"` — full-bleed PNG (for Mermaid/Chart.js slides) |
| `html` | Yes | HTML content for the slide body (used for rendering + annotation targets) |
| `css` | No | Additional CSS rules (injected into `<style>`) |
| `shapes_js` | No | rough.js decoration calls (runs after fonts load) |
| `annotations_js` | No | rough-notation calls (runs after fonts load) |
| `accent_index` | No | Force accent color (0-5). Auto-rotates by slide index if omitted |
| `elements` | No | Array of native PPTX elements (textbox/image). Only used when `render_mode` is `"dual"` |

### Style Fields

| Field | Description |
|-------|-------------|
| `bg` | Background color hex |
| `text` | Primary text color hex |
| `text_muted` | Secondary text color hex |
| `roughness` | rough.js roughness value (0-3) |
| `stroke_width` | rough.js default stroke width |
| `font_map` | Optional font overrides: `{ heading, body, annotation, cjk }` |

**Font mapping** (defaults if `font_map` not specified):

| Category | HTML Font | PowerPoint Default |
|----------|-----------|-------------------|
| heading | Caveat | Ink Free |
| body | Patrick Hand | Ink Free |
| annotation | Kalam | Ink Free |
| cjk | Noto Sans TC | Microsoft JhengHei |

### Element Schema

Elements are native PPTX shapes placed on top of the background PNG. Coordinates are in pixels (1920×1080 canvas).

**Textbox element:**
```json
{
  "type": "textbox",
  "x": 160, "y": 100, "w": 1600, "h": 200,
  "paragraphs": [
    {
      "text": "Heading text",
      "font": "heading",
      "size": 54,
      "color": "#1864ab",
      "bold": true,
      "italic": false,
      "align": "center",
      "bullet": false,
      "level": 0,
      "space_before": 0,
      "space_after": 8,
      "runs": [
        { "text": "Bold part", "bold": true, "font": "heading", "size": 54, "color": "#1864ab" },
        { "text": " normal part", "font": "body", "size": 32 }
      ]
    }
  ]
}
```

- `font`: one of `"heading"`, `"body"`, `"annotation"` — maps to `style.font_map`
- `runs`: optional array for mixed formatting within a paragraph. If omitted, the paragraph's top-level text/font/size/color/bold/italic are used as a single run
- `bullet`: set `true` for bullet point paragraphs
- `level`: bullet indent level (0-based)
- CJK text is automatically handled via `a:ea` typeface on every run

**Image element:**
```json
{
  "type": "image",
  "x": 900, "y": 200, "w": 800, "h": 500,
  "src": "path/to/image.png",
  "label": "[Insert experiment screenshot]",
  "bg_color": "#d0ebff",
  "bg_opacity": 0.3,
  "border_color": "#1971c2",
  "icon": "📷",
  "label_color": "#495057"
}
```

- If `src` exists as a file, inserts the image
- If `src` is null or file doesn't exist, creates a placeholder rectangle with `label` text
- `bg_color`: optional hex color for tinted background fill (makes placeholders look intentional)
- `bg_opacity`: fill opacity 0.0-1.0 (default 0.3)
- `border_color`: optional hex for border color (falls back to style accent, then gray)
- `icon`: optional emoji rendered large above the label text (e.g., "📷", "📊")
- `label_color`: optional hex for label text color (default "#868e96")

### Design Principles

1. **1 slide = 1 message**: Each slide conveys exactly one idea
2. **Never repeat layouts**: Vary structure across slides — mix grids, flex, centering, asymmetric
3. **Typography contrast**: Headings 64-96px (Caveat), body 28-36px (Patrick Hand)
4. **Max 3-5 bullets** per slide — prefer visual over text
5. **Vary decorations**: Different rough.js patterns per slide — frames, corners, underlines, circles
6. **Whitespace is design**: Don't fill every pixel — let content breathe
7. **Accent colors rotate**: Let the palette cycle naturally, or override for emphasis
8. **Follow style preset**: Respect the chosen preset's roughness, strokeWidth, and decoration guidance

### Style-Aware Decorations

When generating `shapes_js`, follow the decoration guidance from the chosen style preset in `STYLE_PRESETS.md`. For example:
- **Clean Sketch**: Thin strokes (1.5-2), max 2 shapes, 40%+ whitespace
- **Bold Marker**: Thick strokes (3-4), hachure fills, 3-5 shapes
- **Notebook**: Grid dots, sticky-note colors, heavy annotations
- **Blackboard**: High roughness (3+), bright stroke colors on dark bg
- **Blueprint**: Low roughness (1.0), all-blue tones, grid lines
- **Watercolor**: Soft fills with low opacity, ellipses over rectangles

## Phase 5: Render + Visual Review

### Render

For dual-layer (editable) PPTX — renders both full preview PNGs and background-only PNGs:
```bash
python .claude/skills/excalidraw-slides/render_slides.py \
  --deck deck.json \
  --outdir slides_tmp/ \
  --mode both
```

For image-only PPTX (legacy behavior):
```bash
python .claude/skills/excalidraw-slides/render_slides.py \
  --deck deck.json \
  --outdir slides_tmp/
```

### Visual Review

After rendering, read each `slide_XX.png` (full preview) with the Read tool to visually check for:
- Text overflow or clipping
- Overlapping elements
- Empty/blank areas that should have content
- Decoration covering text (z-index issues)
- Font rendering issues

If any issues are found, fix the slide JSON and re-render only the affected slides.

## Phase 6: Assemble PPTX

### Dual-layer mode (editable text + hand-drawn background):
```bash
python .claude/skills/excalidraw-slides/gen_pptx.py \
  --deck deck.json \
  --imgdir slides_tmp/ \
  --output "Presentation.pptx"
```

### Legacy image-only mode:
```bash
python .claude/skills/excalidraw-slides/gen_pptx.py \
  --images slides_tmp/slide_01.png slides_tmp/slide_02.png ... \
  --output "Presentation.pptx"
```

### Choosing render_mode per slide

- Use `"dual"` (default) for slides with text/bullet content — user can edit text in PowerPoint
- Use `"image_only"` for Mermaid diagrams, Chart.js charts, or complex visual slides — these are baked into the PNG since they can't be reproduced natively in PPTX

## Output

Final deliverable: `{title}.pptx` in the working directory.

- For CJK text (Chinese/Japanese/Korean), fonts include Noto Sans TC — works automatically

- Read DESIGN_SYSTEM.md for the complete layout patterns library and decoration cookbook
